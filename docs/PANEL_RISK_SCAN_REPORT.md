## 3. API 耦合风险

### 问题描述
`/api/data` 返回 stats + records，使用不同缓存键

### 缓存键
| 数据类型 | 缓存键 | TTL |
|---------|-------|-----|
| stats | stats:v27:2026146:{hash} | 30s |
| records | aggregated_live:v27 | 30s |

### 数据不同步场景
```
时间 T0: API 调用 /api/data
  → stats 缓存: stats:v27:2026146:abc123
  → records 缓存: aggregated_live:v27

时间 T0+5s: 后台更新 aggregated_live.jsonl（新记录写入）

时间 T0+10s: API 调用 /api/data
  → stats 从缓存返回（旧数据，hash=abc123）
  → records 从缓存返回（新数据，已更新）
  → 结果：stats 和 records 不一致
```

### 风险等级
**MEDIUM**

### 风险窗口
30秒缓存 TTL

### 修复建议
- 选项A：统一 TTL，使用相同缓存失效机制
- 选项B：stats 计算时直接读取 records，不单独缓存
- 选项C：增加版本号机制，records 更新时递增版本号
